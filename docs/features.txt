High availability for checkout and check-in (SLA 99.9%+).

Strong fraud protection & ticket authenticity.

3. Data storage & persistence
Primary relational DB: PostgreSQL (one per service or schemas per tenant early on). Use logical separation for multi-tenancy:

Option A (recommended at scale): Database-per-tenant for strong isolation for big organisers; database-per-tenant can be expensive.

Option B (recommended for MVP): Shared DB with tenant_id column + row-level security (RLS) in Postgres.

Cache: Redis for sessions, locks, seat holds, rate-limiting, token storage.

Message broker: Kafka (high throughput) or RabbitMQ for event-driven async processing (reservations release, emails).

Search: ElasticSearch / Opensearch.

Object storage: S3-compatible (AWS S3, or GCP Cloud Storage, or DigitalOcean Spaces).

Analytics data lake: Parquet files on S3 + Snowflake/BigQuery or ClickHouse for real-time analytics.

Time-series / monitoring: Prometheus + Grafana.

Logs: Centralized log storage (ELK/Opensearch stack).

6. Sequence flows (high-level)
A. Purchase (checkout) flow
Client requests GET /events/{id} to show tickets.

User selects ticket(s); client calls POST /checkout with idempotency key.

Checkout service calls Inventory service to hold seats/tickets (place hold with TTL).

Checkout service creates order in DB with status pending.

If payment method MPesa: Payments service initiates MPesa express push; user completes on phone.

Payment gateway sends webhook -> Payments service validates signature -> updates payments -> notifies Ticketing service.

Ticketing service marks order paid, issues tickets (generate QR), saves to DB, triggers email & SMS via Notification service.

Hold releases for abandoned idempotent time; scheduled worker cleans up.

Important: All operations that touch ticket inventory must be transactional or guarded by distributed locks (Redis locks) to avoid oversell.

B. Check-in flow (scanner app)
Staff opens scanner app (auth). App fetches pre-signed manifest & latest sync token.

Scanner scans QR, POST /scanner/scan -> Check-in service validates ticket state & mark checked-in with atomic update.

Response returns attendee info & gate instructions.

If offline, scanner stores scans locally and syncs with server when network available. Server performs reconciliation and flags duplicates.

C. Resale flow
Seller lists ticket: POST /resale/list -> platform verifies ticket is transferable & locks it (status listed_locked).

Buyer purchases via checkout -> payment processing as normal.

On success, ticket ownership updated, listing closed, seller receives payout per fee rules, buyer gets new QR (or transfer recorded).

7. Concurrency, correctness & anti-oversell
Use optimistic locking (version column) or pessimistic locks (Redis distributed locks) when changing inventory.

Implement seat-hold TTLs (e.g., 10 minutes) and background release job.

Use idempotency keys for checkout & payment endpoints.

Generate signed QR payloads (HMAC with server secret) and verify on scan (QR contains token + signature + expiry). Consider rotating signing keys.

Duplicate-scan detection: maintain Redis set of recently-scanned ticket IDs with TTL.

8. Offline-first & resilience for onsite check-ins
Scanner app uses local SQLite/Realm to store scanned entries; sync strategy:

Write-through local queue + background worker to send scans to POST /scanner/sync.

Server validates bulk uploads and returns acknowledgement with resolved states (accepted, duplicate, error).

For issuance fallback, allow offline-issue tokens: pre-generate signed single-use tokens for box-office printing which can later be reconciled.
